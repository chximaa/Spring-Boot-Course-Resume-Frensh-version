PARTIE J1 🏗️

Introduction à l'Architecture J2EE

Architecture multicouche standardisée pour les applications d'entreprise — la base de tout développement Java Web.

Concept de base : Architecture n-tiers

J2EE propose une architecture multicouche (n-tier) standardisée conçue pour les applications d'entreprise. L'objectif principal est la séparation des responsabilités : chaque couche a un rôle précis et bien délimité.

📖 Définition — Architecture J2EE

J2EE (Java 2 Enterprise Edition) est une plateforme Java standardisée pour le développement d'applications web d'entreprise. Elle repose sur un serveur d'application Tomcat (Conteneur de Servlets) et une architecture en trois couches distinctes.

Les 3 couches de l'Architecture J2EE

🌐

Couche WEB

Rôle : Gère l'interaction utilisateur.

Technologies : Servlet (Contrôleur), JSP (Vue)

C'est le point d'entrée de toutes les requêtes HTTP venant du navigateur client.

🧠

Couche Métier

Rôle : Contient les règles de gestion.

Technologies : Services (EJB, Spring)

Toute la logique applicative réside ici — calculs, validations, règles business.

🗄️

Couche DAO

Rôle : Gère l'accès aux données (Persistance).

Technologies : Hibernate, JDBC

Abstraction de la base de données — c'est ici que se trouvent les requêtes SQL.

PARTIE J2 🔄

Le Modèle MVC (Modèle-Vue-Contrôleur)

Le cœur du design pattern de la couche Web J2EE — la séparation entre données, affichage et logique.

Les 3 composants du MVC

🎮

Le Contrôleur (Servlet)

C'est le "chef d'orchestre".

Il reçoit la requête HTTP du client, appelle la logique métier et choisit la page suivante à afficher.

Classe Java : ControllerServlet.java

📦

Le Modèle (JavaBeans)

Ce sont les objets de transport de données.

Exemples : un objet Produit ou Utilisateur avec attributs privés et getters/setters.

Classe Java : Produit.java

🖼️

La Vue (JSP)

C'est la page HTML dynamique.

Elle affiche les données contenues dans le Modèle. En J2EE moderne, on utilise JSTL et l'Expression Language (EL).

Fichier : produits.jsp

PARTIE J4 🔌

La Couche DAO — JDBC & Singleton Connection

Le patron DAO (Data Access Object) et la connexion JDBC via le pattern Singleton.

Qu'est-ce que le patron DAO ?

📖 DAO = Data Access Object

Le DAO est un design pattern qui fait l'interface entre l'application et la base de données, afin de masquer la complexité du SQL. On manipule des objets Java au lieu d'écrire du SQL directement dans le code métier.

AvantageExplication
AbstractionOn manipule des objets Java, pas du SQL directement
Séparation Interface/ImplémentationInterface = le contrat (QUOI), Implémentation = le code JDBC (COMMENT)
Indépendance BDDChanger de MySQL à Oracle ? On ne modifie que le DAO
TestabilitéPeut être mocké pour les tests unitaires

Étape 1 — La Connexion Singleton

Le Singleton garantit qu'une seule instance de connexion MySQL existe dans toute l'application. Ouvrir une nouvelle connexion à chaque requête ferait saturer le serveur MySQL.

java dao/SingletonConnection.java — Annoté ligne par ligne 
package com.gestion.dao;

import java.sql.Connection;
import java.sql.DriverManager;

public class SingletonConnection {

    // 1. Instance UNIQUE partagée par tous les utilisateurs (static)
    private static Connection connection;

    // 2. Bloc statique : s'exécute UNE SEULE FOIS au chargement de la classe
    static {
        try {
            // 3. Charger le pilote JDBC MySQL en mémoire
            Class.forName("com.mysql.cj.jdbc.Driver");

            // 4. Établir la connexion (protocole://hôte:port/base, user, mdp)
            connection = DriverManager.getConnection(
                "jdbc:mysql://localhost:3306/db_produits",
                "root",
                "admin"
            );
            System.out.println("Connexion réussie à MySQL !");

        } catch (Exception e) {
            // 5. Si erreur : afficher sans planter tout le serveur
            e.printStackTrace();
        }
    }

    // 6. Point d'accès unique — appelé par le DAO pour obtenir la connexion
    public static Connection getConnection() {
        return connection;
    }
}

Étape 3 — L'Implémentation JDBC

java dao/ProduitDaoImpl.java — La méthode save() annotée 
public class ProduitDaoImpl implements IProduitDao {

    @Override
    public Produit save(Produit p) {

        // Récupère la connexion existante (Singleton)
        Connection conn = SingletonConnection.getConnection();

        try {
            // PreparedStatement : sécurisé contre les Injections SQL (utilise "?")
            PreparedStatement ps = conn.prepareStatement(
                "INSERT INTO PRODUITS(DESIGNATION, PRIX, QUANTITE) VALUES(?,?,?)"
            );

            // "Mapping" : Objet Java → Paramètres SQL
            ps.setString(1, p.getDesignation());   // 1er ?
            ps.setDouble(2, p.getPrix());           // 2e ?
            ps.setInt(3, p.getQuantite());        // 3e ?

            // Exécuter → génère INSERT INTO PRODUITS VALUES(...);
            ps.executeUpdate();

        } catch (SQLException e) {
            e.printStackTrace();  // Affiche l'erreur sans crasher Tomcat
        }
        return p;
    }

    @Override
    public List<Produit> produitsParMC(String mc) {
        List<Produit> liste = new ArrayList<>();
        try {
            PreparedStatement ps = SingletonConnection.getConnection()
                .prepareStatement("SELECT * FROM PRODUITS WHERE DESIGNATION LIKE ?");
            ps.setString(1, "%" + mc + "%");
            ResultSet rs = ps.executeQuery();
            while (rs.next()) {
                liste.add(new Produit(
                    rs.getString("DESIGNATION"),
                    rs.getDouble("PRIX"),
                    rs.getInt("QUANTITE")
                ));
            }
        } catch (SQLException e) { e.printStackTrace(); }
        return liste;
    }
}

🔍 executeUpdate vs executeQuery

MéthodeQuand l'utiliserSQL correspondantRetour
executeUpdate()Modifier la baseINSERT, UPDATE, DELETENombre de lignes affectées (int)
executeQuery()Lire la baseSELECTResultSet (objet de résultats)
📌 Résumé — Couche DAO
  • Singleton = une seule connexion MySQL pour toute l'application (performance)
  • Interface = définit le contrat (findAll, save, delete...) sans le code
  • PreparedStatement = requêtes paramétrées avec ? → protection contre les injections SQL
  • executeUpdate() pour les modifications, executeQuery() pour les lectures
PARTIE J5 🫘

Le Modèle — JavaBeans

Les objets qui transportent les données entre les couches de l'application J2EE.

Définition et Caractéristiques

📖 JavaBean

Un JavaBean est un composant logiciel réutilisable servant à encapsuler plusieurs objets en un seul — souvent une ligne de base de données. Exemple : un objet Produit représente une ligne de la table PRODUITS.

🔒

Attributs Privés

Tous les attributs sont private. Encapsulation = personne ne peut les modifier directement depuis l'extérieur.

🏗️

Constructeur vide

Doit posséder un constructeur sans argument (par défaut). Requis par les frameworks (JSTL, JSP, Hibernate).

📡

Sérialisable

Doit implémenter java.io.Serializable pour pouvoir circuler sur le réseau entre serveurs.

🔧

Getters & Setters

getNom() = Accesseur (lecture). setNom(String n) = Mutateur (écriture). La JSP utilise les Getters via ${p.nom}.

java model/Produit.java — JavaBean complet 
package com.gestion.model;

import java.io.Serializable;

// Sérialisable : peut voyager sur le réseau entre serveurs
public class Produit implements Serializable {

    // RÈGLE 1 : Attributs PRIVÉS
    private Long   id;
    private String designation;
    private double prix;
    private int    quantite;

    // RÈGLE 2 : Constructeur VIDE (obligatoire)
    public Produit() {}

    // Constructeur avec arguments (pratique)
    public Produit(String designation, double prix, int quantite) {
        this.designation = designation;
        this.prix = prix;
        this.quantite = quantite;
    }

    // RÈGLE 3 : Getters et Setters (générés par Eclipse : Alt+Insert)
    public Long   getId()               { return id; }
    public void   setId(Long id)          { this.id = id; }

    public String getDesignation()       { return designation; }
    public void   setDesignation(String d){ this.designation = d; }

    public double getPrix()              { return prix; }
    public void   setPrix(double prix)   { this.prix = prix; }

    public int    getQuantite()          { return quantite; }
    public void   setQuantite(int q)    { this.quantite = q; }
}
🔗
Lien avec la JSP — Comment la Vue lit les données

En JSP, écrire ${p.designation} n'accède pas directement à l'attribut privé. La JSP appelle automatiquement la méthode Getter correspondante : p.getDesignation(). Si tu n'as pas de Getter, tu obtiens une erreur.

PARTIE J6 🎮

Le Contrôleur — La Servlet

Composant Java qui reçoit les requêtes HTTP et orchestre le flux MVC entre le DAO et la JSP.

Rôles de la Servlet

📬

Réceptionniste HTTP

Intercepte toutes les requêtes dont l'URL se termine par .do grâce à @WebServlet("*.do").

🔀

Routeur d'actions

Analyse le chemin de l'URL (/index.do, /chercher.do) et appelle la méthode DAO correspondante.

📦

Transporteur de données

Stocke le résultat dans l'objet request via setAttribute(), puis fait un forward vers la JSP.

java web/ControllerServlet.java — Annoté ligne par ligne 
@WebServlet(name="cs", urlPatterns={"*.do"})
// ↑ Intercepte TOUTES les URLs finissant par .do (/index.do, /chercher.do...)
public class ControllerServlet extends HttpServlet {

    private IProduitDao dao;

    @Override
    public void init() throws ServletException {
        // S'exécute UNE SEULE FOIS au démarrage — parfait pour instancier le DAO
        dao = new ProduitDaoImpl();
    }

    @Override
    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {

        // ÉTAPE 1 : Récupérer le chemin de l'URL (/index.do, /chercher.do...)
        String path = request.getServletPath();

        // ÉTAPE 2 : Routage via if/else selon l'action demandée
        if (path.equals("/index.do")) {
            // Afficher TOUS les produits (mot-clé vide = tout retourner)
            List<Produit> prods = dao.produitsParMC("");

            // ÉTAPE 3 : Stocker dans request (le "coffre" de la requête)
            request.setAttribute("produits", prods);

            // ÉTAPE 4 : Forward vers la Vue JSP
            request.getRequestDispatcher("produits.jsp").forward(request, response);

        } else if (path.equals("/chercher.do")) {
            // Récupérer la valeur saisie dans le champ HTML name="motCle"
            String mc = request.getParameter("motCle");
            List<Produit> prods = dao.produitsParMC(mc);
            request.setAttribute("produits", prods);
            request.getRequestDispatcher("produits.jsp").forward(request, response);
        }
    }

    @Override
    protected void doPost(...) {
        doGet(request, response); // Déléguer POST vers doGet
    }
}

📝
Règle d'or de la Servlet

Il ne doit y avoir aucune ligne SQL dans la Servlet (rôle du DAO) et aucune ligne HTML (rôle de la JSP). La Servlet se contente de : recevoir la requête, décider quelle méthode du DAO appeler, et transmettre le résultat à la JSP.

PARTIE J7 🖼️

La Vue — JSP & JSTL

JavaServer Pages avec la bibliothèque JSTL pour générer du HTML dynamique sans écrire de Java brut.

JSP vs JSTL — Pourquoi éviter les Scriptlets ?

❌

Ancien style — Scriptlets

Écrire du Java dans du HTML avec <% ... %>

Problème : Code illisible, difficile à maintenir, dangereux pour les designers web.

✅

Moderne — JSTL + EL

Balises HTML-like avec l'Expression Language ${...}

Avantage : Lisible, maintenable, séparation propre entre Java et HTML.

jsp produits.jsp — Vue complète annotée 
<!-- Directive 1 : Configuration de la page -->
<%@ page language="java" contentType="text/html; charset=UTF-8" pageEncoding="UTF-8"%>

<!-- Directive 2 : Importer la bibliothèque JSTL Core (prefix="c") -->
<%@ taglib uri="http://java.sun.com/jsp/jstl/core" prefix="c" %>

<html>
<head><title>Ma Boutique</title></head>
<body>

  <h1>Liste des Produits</h1>

  <!-- Formulaire : action="chercher.do" → appelle /chercher.do dans la Servlet -->
  <!--              name="motCle" → récupéré par request.getParameter("motCle") -->
  <form action="chercher.do" method="get">
    Mot-clé : <input type="text" name="motCle">
    <button type="submit">Rechercher</button>
  </form>

  <table border="1">
    <tr><th>ID</th><th>Désignation</th><th>Prix</th></tr>

    <!-- items="${produits}" = la liste stockée par request.setAttribute("produits", ...) -->
    <!-- var="p" = à chaque tour de boucle, "p" est un objet Produit              -->
    <c:forEach items="${produits}" var="p">
      <tr>
        <td>${p.id}</td>           <!-- appelle p.getId() -->
        <td>${p.designation}</td> <!-- appelle p.getDesignation() -->
        <td>${p.prix} €</td>      <!-- appelle p.getPrix() -->
      </tr>
    </c:forEach>

  </table>
</body>
</html>

  • JSP = page HTML + code Java dynamique côté serveur
  • <%@ taglib prefix="c" %> importe la bibliothèque JSTL Core

  • La JSP ignore l'existence de MySQL ou du DAO — elle ne fait qu'afficher

Questions basées sur le cours J2EE — Architecture, DAO, Servlets et JSP

Partie A — Questions à Choix Multiple
Question J01

Apache Tomcat est techniquement appelé :

  • A) Un serveur de base de données
  • B) Un serveur FTP
  • C) Un Conteneur de Servlets (Web Container)
  • D) Un compilateur Java
✅ Réponse : C — Tomcat est un Conteneur de Servlets. Il a 4 rôles : Réceptionniste HTTP (écoute sur le port 8080), Gestionnaire de vie (appelle init, doGet, destroy), Chef de chantier (multi-threading), et Bibliothécaire (sécurise WEB-INF, charge les .jar).
Question J02

Dans le patron DAO, pourquoi utilise-t-on un PreparedStatement avec des ? plutôt que de construire la requête SQL avec des concaténations de chaînes ?

  • A) Pour écrire moins de code Java
  • B) Pour se protéger contre les injections SQL
  • C) Car MySQL ne comprend pas la concaténation
  • D) Pour améliorer l'affichage des résultats
✅ Réponse : B — Le PreparedStatement avec des ? est une mesure de sécurité indispensable contre les Injections SQL. Si un utilisateur malveillant tape ' OR '1'='1 dans un champ, la concaténation directe rendrait la requête dangereuse. Les ? traitent la valeur comme un simple paramètre, jamais comme du SQL exécutable.
Question J03

Quelle méthode de la Servlet est appelée UNE SEULE FOIS au démarrage, et est idéale pour instancier le DAO ?

  • A) doGet()
  • B) init()
  • C) doPost()
  • D) destroy()
✅ Réponse : B — La méthode init() fait partie du cycle de vie de la Servlet géré par Tomcat. Elle s'exécute exactement une fois lors du premier chargement de la Servlet. C'est l'endroit idéal pour instancier le DAO : dao = new ProduitDaoImpl();. doGet() et doPost() sont appelés à chaque requête. destroy() s'exécute à l'arrêt.
Question J04

En JSP, que fait exactement ${p.designation} ?

  • A) Lit directement l'attribut privé designation de l'objet
  • B) Appelle automatiquement la méthode p.getDesignation()
  • C) Exécute une requête SQL SELECT
  • D) Affiche le nom de la classe Java
✅ Réponse : B — L'Expression Language (EL) de JSP appelle automatiquement le Getter correspondant. ${p.designation} = p.getDesignation(). C'est pourquoi les JavaBeans doivent avoir des Getters — sans eux, la JSP ne peut pas lire les données et génère une erreur.
Question J05

Quelle est la différence entre executeUpdate() et executeQuery() en JDBC ?

  • A) Il n'y a aucune différence, les deux font la même chose
  • B) executeUpdate() est pour les SELECT, executeQuery() pour les INSERT
  • C) executeUpdate() pour les modifications (INSERT/UPDATE/DELETE), executeQuery() pour les lectures (SELECT)
  • D) Les deux retournent un ResultSet
✅ Réponse : C — executeUpdate() est utilisé pour les requêtes qui modifient la base (INSERT, UPDATE, DELETE) et retourne un int (nombre de lignes affectées). executeQuery() est utilisé pour les requêtes de lecture (SELECT) et retourne un ResultSet contenant les lignes résultantes.
Question J06

Quel est le rôle de request.setAttribute("produits", prods) dans la Servlet ?

  • A) Envoie les données directement au navigateur client
  • B) Sauvegarde les données dans la base MySQL
  • C) Stocke la liste dans l'objet request pour que la JSP puisse y accéder via ${produits}
  • D) Crée une session permanente pour l'utilisateur
✅ Réponse : C — C'est le cœur du MVC. La Servlet stocke le résultat du DAO dans un "coffre" nommé "produits" dans l'objet request. Ensuite, via forward(), la JSP reçoit cet objet et peut accéder à la liste avec ${produits} dans une boucle <c:forEach>.

PARTIE 01 🏛️

L'Architecture Monolithique

La façon traditionnelle de construire une application — tout en un seul bloc.

C'est quoi un Monolithe ?

Imagine que tu construis une maison, mais au lieu d'avoir des pièces séparées (chambre, cuisine, salon, salle de bain), toutes les pièces sont dans une seule grande salle ouverte. Tout est mélangé, tout est au même endroit. C'est exactement ça, un monolithe.

📖 Définition officielle

Une architecture monolithique est une application construite comme une seule et unique unité indivisible. Tout le code (routage, fonctionnalités, logique métier) est dans le même projet, compilé ensemble, et déployé sur un seul serveur.

Les 4 composants d'un Monolithe

🌐

1. Les Clients

Ce que c'est : Les utilisateurs qui envoient des requêtes.

Exemples : Un navigateur web, une application mobile, une autre API.

Ils envoient tous leurs requêtes vers un seul point d'entrée.

🖥️

2. Le Serveur Monolithique

Ce que c'est : Le cœur du système. Tout le code est là.

Feature A (Utilisateurs) + Feature B (Produits) + Feature C (Paiements) partagent la même mémoire et le même processeur.

🔀

3. Le Router (Routeur)

Ce que c'est : Le "réceptionniste" de l'application.

Il reçoit toutes les requêtes et les dirige vers la bonne fonctionnalité dans le code.

🗄️

4. Une seule Base de Données

Ce que c'est : Une seule base de données partagée par tout le monde.

Feature A, B et C lisent et écrivent toutes dans la même base. Très fort couplage.

✅ Avantages & ❌ Limites du Monolithe

CaractéristiqueAvantage/LimiteDétail
Démarrage✅ Simple au débutUn seul projet, déploiement facile (.jar ou .war)
Tests✅ SimplifiésTout est au même endroit, les tests couvrent tout
Mise à l'échelle❌ DifficileDupliquer tout le monolithe même si une seule feature a besoin de scaling
Maintenance❌ Code SpaghettiModifier Feature A peut accidentellement casser Feature B
Déploiement❌ RisquéUne ligne modifiée → redémarrage TOTAL → interruption de service
Technologie❌ PrisonCoincé avec un seul langage pour tout le projet
📌 Résumé — Chapitre 1
  • Un monolithe = tout en un seul bloc déployable
  • Idéal pour les petits projets, catastrophique pour les grands systèmes
  • 3 problèmes majeurs : difficile à scaler, difficile à maintenir, dangereux à déployer
  • Amazon, Netflix, Uber... tous sont passés du monolithe aux microservices
PARTIE 02 🔬

Les Microservices — La Solution Moderne

Découper l'application en petits services autonomes, indépendants et communicants.

Définition des Microservices

📖 Définition

Une architecture microservices est une approche qui consiste à découper une application en une suite de petits services indépendants, chacun exécutant son propre processus, avec sa propre base de données, et communiquant via des mécanismes légers (généralement HTTP/REST).

Les 3 principes fondamentaux

1
Couplage Faible (Loose Coupling)
Les services se connaissent le moins possible. Un service peut changer son implémentation interne sans que les autres ne le sachent.
🔧 Analogie : Tu peux changer une ampoule sans connaître l'architecture électrique du bâtiment.
2
Cohésion Forte (High Cohesion)
Tout le code lié à un sujet reste ensemble. La logique de facturation est dans le Service Facturation, pas éparpillée partout.
3
Indépendance de Déploiement
Je peux mettre à jour le Service A sans éteindre ou affecter le Service B. Chaque service a son propre cycle de déploiement.

📊 Comparaison complète : Monolithe vs Microservices

CaractéristiqueMonolitheMicroservices
Simplicité initialeSimple à démarrerInfrastructure complexe dès le début
Langage de programmationUn seul langage pour toutPolyglotte — chaque service peut avoir son propre langage
Mise à l'échelleVerticale : acheter un serveur plus puissantHorizontale : dupliquer uniquement le service qui en a besoin
En cas de panneUne boucle infinie plante tout le serveurLa panne est isolée à un seul service
DéploiementRedémarrage complet pour chaque modificationDéploiement indépendant service par service
Base de donnéesUne seule base partagée par tout le mondeUne base de données par service
⚠️
⭐ Règle d'Or — Database per Service

Chaque microservice doit posséder sa propre base de données privée. Service A ne peut JAMAIS accéder directement à la base de données de Service B. Ils communiquent uniquement via leurs APIs.

📌 Résumé — Chapitre 2
  • Microservices = un ensemble de petits services autonomes et indépendants
  • Règle d'or : chaque service a SA PROPRE base de données — jamais partagée
  • 3 principes : Couplage faible, Cohésion forte, Déploiement indépendant
  • Évite le "Monolithe Distribué" — ne chaîne pas les services de façon synchrone

Le Patron DAO Spring — Injection de Dépendances

Séparer l'accès aux données de la logique métier — version Spring avec @Repository et IoC.

Injection par Constructeur (méthode recommandée)

💉
Injection de Dépendances (Dependency Injection)

Au lieu de créer toi-même l'objet avec new ProductDaoImpl(), tu laisses Spring le créer et te le donner. C'est le principe d'Inversion de Contrôle (IoC). Spring gère la création et le cycle de vie des objets (appelés Beans).

java dao/ProductDaoImpl.java — @Repository Spring 
@Repository
// ↑ Dit à Spring : "Cette classe est dédiée à l'accès aux données"
// ↑ Spring crée automatiquement une instance (Bean) et l'injecte où nécessaire
public class ProductDaoImpl implements ProductDao {

    private static List<Product> products = new ArrayList<>(Arrays.asList(
        new Product(1, "Ordinateur portable", 350),
        new Product(2, "Aspirateur Robot",    500),
        new Product(3, "Table de Ping Pong", 750)
    ));

    @Override
    public List<Product> findAll() { return products; }

    @Override
    public Product findById(int id) {
        for (Product p : products)
            if (p.getId() == id) return p;
        return null;
    }
}

🔑 JPA vs Hibernate — La différence essentielle

📋

JPA = La Spécification (les règles)

Quel est le principal problème de l'architecture monolithique pour les grands systèmes ?

  • A) Elle est trop simple à construire

  • C) Elle nécessite toujours un serveur externe séparé

Que combine exactement l'annotation @RestController ?

  • A) @Service + @Repository
  • B) @Controller + @ResponseBody
  • C) @Entity + @Table
  • D) @Component + @Configuration
✅ Réponse : B — @Controller gère les requêtes HTTP. @ResponseBody sérialise le retour directement en JSON. @RestController = les deux en une seule annotation.

Quelle annotation utilises-tu pour extraire une valeur depuis l'URL, par exemple /Produits/42 ?

Quelle est la relation exacte entre JPA et Hibernate ?

  • B) JPA est la spécification (les règles), Hibernate est une implémentation de ces règles
  • C) Ce sont exactement la même chose
  • D) Hibernate est le successeur de JPA
✅ Réponse : B — JPA = Code de la Route (règles). Hibernate = la Voiture (exécute les règles). Spring Boot utilise JPA comme API standard, mais c'est Hibernate qui génère le SQL en arrière-plan.
