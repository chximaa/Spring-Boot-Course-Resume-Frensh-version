PARTIE 01 🏛️

L'Architecture Monolithique

La façon traditionnelle de construire une application — tout en un seul bloc.

C'est quoi un Monolithe ?

Imagine que tu construis une maison, mais au lieu d'avoir des pièces séparées (chambre, cuisine, salon, salle de bain), toutes les pièces sont dans une seule grande salle ouverte. Tout est mélangé, tout est au même endroit. C'est exactement ça, un monolithe.

📖 Définition officielle

Une architecture monolithique est une application construite comme une seule et unique unité indivisible. Tout le code (routage, fonctionnalités, logique métier) est dans le même projet, compilé ensemble, et déployé sur un seul serveur.

Les 4 composants d'un Monolithe

🌐

1. Les Clients

Ce que c'est : Les utilisateurs qui envoient des requêtes.

Exemples : Un navigateur web, une application mobile, une autre API.

Ils envoient tous leurs requêtes vers un seul point d'entrée.

🖥️

2. Le Serveur Monolithique

Ce que c'est : Le cœur du système. Tout le code est là.

Feature A (Utilisateurs) + Feature B (Produits) + Feature C (Paiements) partagent la même mémoire et le même processeur.

🔀

3. Le Router (Routeur)

Ce que c'est : Le "réceptionniste" de l'application.

Il reçoit toutes les requêtes et les dirige vers la bonne fonctionnalité dans le code.

🗄️

4. Une seule Base de Données

Ce que c'est : Une seule base de données partagée par tout le monde.

Feature A, B et C lisent et écrivent toutes dans la même base. Très fort couplage.

Organisation en équipes (couches)

Dans un monolithe, les équipes sont organisées par couche technique et non par fonctionnalité. Cela crée une forte interdépendance :

👥
Spécialisation par couche

L'équipe Front-End dépend du Back-End pour les données. Le Back-End dépend de la base de données. Si une équipe est bloquée, tout le monde attend. Dans un monolithe, tout est étroitement intégré.

✅ Les avantages du Monolithe

AvantagePourquoi c'est bien ?Exemple concret
Simplicité au débutUn seul projet à créer, configurer et lancerUn étudiant peut créer une app complète en une journée
Déploiement facileUn seul fichier (.jar ou .war) à copier sur le serveurComme copier un seul fichier sur une clé USB
Tests simplifiésTout est au même endroit, les tests couvrent toutUn seul test peut vérifier Feature A ET Feature B

❌ Les limites du Monolithe

ProblèmeCe qui se passe concrètement
Mise à l'échelle difficileSi la Feature A reçoit beaucoup de trafic, tu dois dupliquer tout le monolithe, même Features B et C qui n'en ont pas besoin. Énorme gaspillage de ressources.
Code SpaghettiAvec le temps, le code devient gigantesque et emmêlé. Modifier Feature A peut accidentellement casser Feature B.
Déploiements risquésChanger UNE ligne de code dans Feature C oblige à redémarrer TOUTE l'application → interruption de service pour tout le monde.
Prison technologiqueTu es coincé avec un seul langage pour tout. Impossible d'écrire Feature A en Java et Feature B en Python.
Cycles lentsPlus la base de code grandit, plus les déploiements deviennent longs et risqués.

📖 L'exemple réel d'Amazon

🛒
Amazon — de monolithe à microservices

Amazon était au départ une seule application monolithique. Résultat : intégration difficile, des semaines entières pour fusionner les modifications de code, des conflits permanents entre développeurs. La solution : adopter une architecture microservices → chaque service a sa propre responsabilité, accessible via une API web, les équipes travaillent indépendamment.

📌 Résumé — Chapitre 1
  • Un monolithe = tout en un seul bloc déployable
  • Idéal pour les petits projets, catastrophique pour les grands systèmes
  • 3 problèmes majeurs : difficile à scaler, difficile à maintenir, dangereux à déployer
  • Les équipes sont interdépendantes → ralentissement général
  • Amazon, Netflix, Uber... tous sont passés du monolithe aux microservices
PARTIE 02 🔬

Les Microservices — La Solution Moderne

Découper l'application en petits services autonomes, indépendants et communicants.

Définition des Microservices

📖 Définition

Une architecture microservices est une approche qui consiste à découper une application en une suite de petits services indépendants, chacun exécutant son propre processus, avec sa propre base de données, et communiquant via des mécanismes légers (généralement HTTP/REST).

Au lieu d'une seule grande salle (monolithe), imagine maintenant des appartements séparés. Chaque appartement (service) a sa propre entrée, ses propres meubles, ses propres règles. Si un appartement prend feu, les autres ne sont pas affectés.

Anatomie d'un Microservice

Chaque microservice est une unité totalement autonome. Il possède :

🧠

Sa propre logique métier

Il ne gère qu'un seul domaine. Ex : le service Produits ne fait que gérer les produits. Jamais les commandes.

📚

Ses propres librairies

Chaque service peut utiliser les dépendances qui lui conviennent le mieux, indépendamment des autres services.

🌐

Son propre serveur web

Un serveur Tomcat, Jetty ou Undertow intégré. Chaque service tourne sur un port différent.

🗄️

Sa propre base de données ⭐

C'est la règle d'or. Le service Produits a sa BDD. Le service Commandes a la sienne. Jamais de partage direct de BDD.

⚠️
⭐ Règle d'Or — Database per Service

Chaque microservice doit posséder sa propre base de données privée. Service A ne peut JAMAIS accéder directement à la base de données de Service B. Ils communiquent uniquement via leurs APIs. C'est ce qui garantit leur vraie indépendance.

Les 3 principes fondamentaux

1
Couplage Faible (Loose Coupling)
Les services se connaissent le moins possible. Un service peut changer son implémentation interne sans que les autres ne le sachent.
🔧 Analogie : Tu peux changer une ampoule sans connaître l'architecture électrique du bâtiment.
2
Cohésion Forte (High Cohesion)
Tout le code lié à un sujet reste ensemble. La logique de facturation est dans le Service Facturation, pas éparpillée partout.
📚 Analogie : Tous les ingrédients d'une pizza sont dans la section pizza du livre de recettes.
3
Indépendance de Déploiement
Je peux mettre à jour le Service A sans éteindre ou affecter le Service B. Chaque service a son propre cycle de déploiement.
🏠 Analogie : Rénover un appartement n'oblige pas les voisins à déménager.

📊 Comparaison complète : Monolithe vs Microservices

CaractéristiqueMonolitheMicroservices

🕷️ Le Piège : Le "Monolithe Distribué"

🕸️
Attention — Ne tombe pas dans ce piège !

Si Service A appelle toujours Service B, qui appelle toujours Service C, et que C tombe en panne → tout tombe en panne. Tu as créé un système complexe et lent, sans gagner en robustesse. C'est le "Monolithe Distribué". Règle : minimiser les appels synchrones entre services.

📌 Résumé — Chapitre 2
  • Microservices = un ensemble de petits services autonomes et indépendants
  • Règle d'or : chaque service a SA PROPRE base de données — jamais partagée
  • 3 principes : Couplage faible, Cohésion forte, Déploiement indépendant
  • Idéal pour les grandes applications et les grandes équipes
  • Évite le "Monolithe Distribué" — ne chaîne pas les services de façon synchrone
PARTIE 03 🌱

Le Framework Spring Boot

L'outil qui facilite la création de microservices Java — zero configuration, démarrage ultra-rapide.

C'est quoi Spring Boot ?

📖 Définition

Spring Boot est un framework Java qui facilite la création d'applications Spring. Il produit une application packagée dans un fichier JAR autonome et exécutable, avec un serveur web (Tomcat) déjà intégré à l'intérieur — plus besoin d'installer un serveur séparément.

😩 Le problème avant Spring Boot

Avec l'ancien Spring ou Java EE, afficher un simple "Hello World" dans un navigateur nécessitait :

  • Installer manuellement un serveur d'application (Tomcat, Glassfish)
  • Écrire et maintenir des dizaines de fichiers XML (web.xml, persistence.xml...)
  • Passer des heures, voire des jours à faire de la "plomberie" avant d'écrire la moindre logique
  • Les erreurs XML ne se détectaient qu'au lancement — debugging cauchemardesque

✨ Les 3 promesses de Spring Boot

⚙️

Promesse 1 : L'Auto-Configuration

Ce que c'est : Spring Boot détecte automatiquement les librairies présentes et les configure pour toi.

Comment ça marche : Tu ajoutes la librairie H2 dans ton projet → Spring Boot crée automatiquement une connexion DataSource. Zéro ligne de configuration !

Annotation clé : @EnableAutoConfiguration — elle parcourt le classpath et dit "Je vois H2, l'utilisateur veut probablement une base de données, je la configure tout seul."

Règle de priorité : Ta propre configuration écrase toujours celle de Spring Boot.

📦

Promesse 2 : Application Autonome

Avant : Tu générais un fichier .war et tu le "déposais" dans un serveur Tomcat déjà installé. L'application allait vers le serveur.

Avec Spring Boot : Le serveur vient dans l'application ! Tout est emballé dans un seul Fat JAR : ton code + toutes les dépendances + Tomcat intégré.

Pour lancer ton application entière : java -jar monapp.jar

🎒

Promesse 3 : Les Starters

Au lieu d'ajouter 30 librairies individuellement, Spring Boot propose des Starters — des kits pré-emballés par fonctionnalité.

1 ligne dans pom.xml → ~30 librairies compatibles installées automatiquement.

📁 Le fichier pom.xml — La carte d'identité du projet

Le pom.xml (Project Object Model) est le fichier de configuration Maven à la racine de ton projet. C'est à la fois le certificat de naissance de ton projet et sa liste de courses.

xml pom.xml — Structure complète annotée 
<!-- ① IDENTITÉ du projet (le GAV : GroupId, ArtifactId, Version) -->
<groupId>com.ecommerce</groupId>         <!-- Nom de ton organisation (domaine inversé) -->
<artifactId>microcommerce</artifactId>   <!-- Nom du projet / du fichier JAR produit -->
<version>0.0.1-SNAPSHOT</version>       <!-- Version actuelle (SNAPSHOT = en développement) -->

<!-- ② HÉRITAGE — Le "DNA" Spring Boot -->
<!-- Ton projet hérite de spring-boot-starter-parent.           -->
<!-- Avantage : il gère automatiquement les versions de TOUTES  -->
<!-- les librairies pour éviter les conflits.                   -->
<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.2.0</version>
</parent>

<!-- ③ DÉPENDANCES — Les librairies nécessaires -->
<dependencies>

  <!-- 1 ligne = ~30 librairies : Tomcat + Jackson + Spring MVC -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <!-- Pas besoin de mettre une version : le Parent gère ça -->
  </dependency>

</dependencies>

<!-- ④ PLUGIN — L'outil qui crée le Fat JAR -->
<build>
  <plugins>
    <plugin>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-maven-plugin</artifactId>
      <!-- Ce plugin emballe TOUT (code + dépendances + Tomcat) -->
      <!-- dans un seul fichier JAR exécutable.                  -->
      <!-- Il permet aussi : mvn spring-boot:run               -->
    </plugin>
  </plugins>
</build>

  • L'Auto-configuration devine tes besoins selon les librairies présentes
  • Fat JAR = ton code + Tomcat + toutes les librairies dans UN seul fichier

PARTIE 05 📡

REST & les Verbes HTTP

Le langage universel de communication entre microservices.

L'analogie du Restaurant

🍽️
Comprendre une API REST avec un restaurant

👤 Le Client (Toi) → Tu veux commander un plat
🍳 La Cuisine (Serveur/Base de données) → C'est là que se trouve la nourriture, mais tu n'as pas le droit d'entrer
🧑‍💼 Le Serveur (L'API REST) → Il prend ta commande (requête), l'apporte en cuisine, et te ramène ton plat (réponse)
Conclusion : L'API REST est le messager entre ton besoin et la donnée brute.

📖 Définition REST

REST (Representational State Transfer) est un style d'architecture qui utilise le protocole HTTP pour échanger des données entre un client et un serveur. Une API REST est une "fenêtre" qu'une application ouvre pour permettre à d'autres logiciels de communiquer avec elle.

📝 Le JSON — Le langage universel des données

Une API REST ne renvoie pas une page HTML. Elle renvoie de la donnée brute au format JSON.

json Exemple de réponse JSON 
{
  "id": 1,
  "nom": "Ordinateur Portable",
  "prix": 350
}
// JSON = format texte léger, lisible par un humain ET par toutes les machines
// (Java, Python, JavaScript, C#... tout le monde comprend JSON)

Verbe HTTPActionÉquivalent SQLExemple concret
GETLIRE — Récupérer des donnéesSELECT"Montre-moi la liste des produits"
POSTCRÉER — Ajouter une nouvelle ressourceINSERT"Ajoute ce nouveau produit"

💡
Principe REST — Ressources et URLs

En REST, tout élément manipulé (Produit, Commande, Utilisateur...) est appelé une Ressource. Chaque ressource a une adresse unique (URL). On agit sur les ressources via les verbes HTTP. Exemple : http://mon-service/produits/12 = le produit numéro 12 de notre service.

PARTIE 06 🕹️

Le Contrôleur & le Retour JSON

Créer le Modèle de données, le Contrôleur REST, et comprendre la magie de Jackson.

📦 Étape 1 — Créer le Modèle de données (Product.java)

📏
Les règles d'or d'un JavaBean / POJO

Pour qu'un objet Java puisse être converti en JSON automatiquement par Jackson, il doit respecter ces règles : 1) Attributs privés (encapsulation) · 2) Constructeur public sans arguments · 3) Getters et Setters pour chaque attribut · 4) (Optionnel) méthode toString()

java model/Product.java — Annoté ligne par ligne 
package com.ecommerce.microcommerce.model;

public class Product {

    // RÈGLE 1 : Attributs PRIVÉS → personne ne peut les modifier directement
    private int id;
    private String nom;    // "nom" = le nom du produit
    private int prix;     // "prix" = le prix du produit

    // RÈGLE 2 : Constructeur SANS arguments (requis par Jackson pour désérialiser)
    public Product() {}

    // Constructeur AVEC tous les champs (pratique pour créer des objets dans le code)
    public Product(int id, String nom, int prix) {
        this.id = id;      // "this.id" = attribut de l'objet, "id" = paramètre
        this.nom = nom;
        this.prix = prix;
    }

    // RÈGLE 3 : Getters (lecture) et Setters (écriture)
    // IntelliJ les génère automatiquement : Alt+Insert → Getter and Setter

    public int getId() { return id; }
    public void setId(int id) { this.id = id; }

    public String getNom() { return nom; }
    public void setNom(String nom) { this.nom = nom; }

    public int getPrix() { return prix; }
    public void setPrix(int prix) { this.prix = prix; }

    // RÈGLE 4 (optionnel) : toString() pour afficher l'objet facilement
    @Override
    public String toString() {
        return "Product{id=" + id + ", nom='" + nom + "', prix=" + prix + "}";
    }
}

🕹️ Étape 2 — Créer le Contrôleur REST

Le Contrôleur est le "cerveau" de ton microservice. Il écoute les requêtes HTTP et renvoie les réponses.

java web/controller/ProductController.java — Explication ligne par ligne 
package com.ecommerce.microcommerce.web.controller;

import com.ecommerce.microcommerce.model.Product;

@RestController
// ┌─ @Controller  → Cette classe gère les requêtes HTTP entrantes
// └─ @ResponseBody → Ce que la méthode retourne est envoyé directement au client
//                    en JSON (pas de page HTML)
public class ProductController {

    // ═══ ENDPOINT 1 ══════════════════════════════════════════════════════
    @GetMapping(value = "/Produits")
    // ↑ Cette méthode répond aux requêtes GET envoyées à /Produits
    public List<Product> listeProduits() {
        // Pour l'instant : liste "en dur". Plus tard, on utilisera le DAO.
        return Arrays.asList(
            new Product(1, "Ordinateur portable", 350),
            new Product(2, "Aspirateur Robot", 500),
            new Product(3, "Table de Ping Pong", 750)
        );
        // Spring voit qu'on retourne une List<Product>
        // Il appelle Jackson qui crée ce JSON : [{...}, {...}, {...}]
    }

    // ═══ ENDPOINT 2 ══════════════════════════════════════════════════════
    @GetMapping(value = "/Produits/{id}")
    // ↑ {id} = partie VARIABLE de l'URL. Ex: /Produits/42 → id = 42
    public Product afficherUnProduit(@PathVariable int id) {
        // @PathVariable = pont entre {id} dans l'URL et la variable Java "id"
        // Spring extrait la valeur de l'URL et la met dans la variable

        // Simulation : on crée un produit factice avec l'id reçu
        return new Product(id, "Aspirateur", 100);
        // Spring retourne → {"id": 42, "nom": "Aspirateur", "prix": 100}
    }
}

⚡ Comment Jackson convertit automatiquement en JSON ?

Quand ta méthode retourne un objet Product, Spring Boot utilise la librairie Jackson pour le convertir en JSON :

Objet Product
En mémoire Java
→
Jackson appelle
les Getters
getId() → "id":1

L'inverse fonctionne aussi (désérialisation) : Jackson lit le JSON entrant et appelle tes Setters pour créer un objet Java.

📌 Résumé — Chapitre 6
  • @RestController = @Controller + @ResponseBody → retour JSON automatique
  • @GetMapping, @PostMapping... lient un verbe HTTP à une méthode Java
  • @PathVariable extrait une valeur depuis l'URL (ex: /Produits/{id})
  • Jackson convertit automatiquement les objets Java ↔ JSON
  • Le Modèle doit être un JavaBean valide (getters/setters/constructeur vide) pour que Jackson fonctionne
PARTIE 07 🗃️

Le Patron DAO

Séparer l'accès aux données de la logique métier — une bonne pratique fondamentale.

C'est quoi un DAO ?

📖 DAO = Data Access Object

Le DAO est un patron de conception (Design Pattern) qui consiste à créer une classe intermédiaire dédiée à l'accès aux données. Le Contrôleur ne va plus directement chercher les données — il demande au DAO de le faire à sa place.

📚
Analogie du Bibliothécaire

Tu ne vas pas toi-même dans la réserve de la bibliothèque chercher un livre. Tu demandes au bibliothécaire (DAO). Il connaît l'organisation du stockage et te ramène ce que tu veux. Toi, tu n'as pas besoin de savoir où les livres sont rangés.

Pourquoi utiliser un DAO ?

AvantageExplication concrète
Séparation des responsabilitésLe Contrôleur gère le HTTP. Le DAO gère les données. Chacun son rôle.
Indépendance de la sourceChanger de MySQL à MongoDB ? Tu ne modifies que le DAO. Le Contrôleur ne change pas.
RéutilisabilitéPlusieurs contrôleurs peuvent utiliser le même DAO.
TestabilitéOn peut tester le Contrôleur sans vraie base de données (via des Mocks).

📝 Étape 1 — L'Interface (Le Contrat)

java dao/ProductDao.java — L'interface (le contrat) 
package com.ecommerce.microcommerce.dao;

public interface ProductDao {

    // Une interface définit QUOI faire, pas COMMENT le faire
    // Quiconque implémente cette interface DOIT fournir ces 3 méthodes

    List<Product> findAll();
    // ↑ Retourne la liste complète de tous les produits

    Product findById(int id);
    // ↑ Trouve un produit par son identifiant unique

    Product save(Product product);
    // ↑ Sauvegarde (ajoute ou met à jour) un produit
}
✨
Pourquoi ces noms précis (findAll, findById, save) ?

Ce sont les conventions de nommage de Spring Data JPA. En respectant ces noms maintenant, Spring sera capable plus tard de générer automatiquement le code SQL correspondant, simplement en lisant le nom des méthodes. C'est la magie qu'on verra au Chapitre 10 !

🔨 Étape 2 — L'Implémentation (La Simulation avec ArrayList)

java dao/ProductDaoImpl.java — L'implémentation simulée 
@Repository
// ↑ Dit à Spring : "Cette classe est dédiée à l'accès aux données"
// ↑ Spring crée automatiquement une instance de cette classe (un Bean)
// ↑ Spring traduit aussi les exceptions BDD en exceptions Java standard
public class ProductDaoImpl implements ProductDao {

    // static = PARTAGÉ par tous. Si l'Utilisateur A ajoute un produit,
    // l'Utilisateur B le verra aussi (tant que le serveur tourne).
    private static List<Product> products = new ArrayList<>(Arrays.asList(
        new Product(1, "Ordinateur portable", 350),
        new Product(2, "Aspirateur Robot",    500),
        new Product(3, "Table de Ping Pong", 750)
    ));

    @Override
    public List<Product> findAll() {
        return products;  // Retourne simplement toute la liste
    }

    @Override
    public Product findById(int id) {
        for (Product p : products) {       // Parcourt toute la liste
            if (p.getId() == id) return p;  // Retourne si l'ID correspond
        }
        return null;  // Pas trouvé → null (on améliorera ça au Chapitre 12)
    }

    @Override
    public Product save(Product product) {
        products.add(product);  // Ajoute le produit à la liste
        return product;          // Retourne le produit sauvegardé
    }
}

🔌 Étape 3 — L'Injection de Dépendances dans le Contrôleur

💉
Injection de Dépendances (Dependency Injection)

Au lieu de créer toi-même l'objet avec new ProductDaoImpl(), tu laisses Spring le créer et te le donner. C'est le principe d'Inversion de Contrôle (IoC). Spring gère la création et le cycle de vie des objets (appelés Beans).

java ProductController.java — Avec injection du DAO 
@RestController
public class ProductController {

    // "final" = une fois initialisé, ne change jamais (sécurité)
    private final ProductDao productDao;

    // INJECTION PAR CONSTRUCTEUR (méthode recommandée)
    // Spring voit ce constructeur → cherche un Bean @Repository
    // qui implémente ProductDao → injecte automatiquement ProductDaoImpl
    public ProductController(ProductDao productDao) {
        this.productDao = productDao;
    }

    @GetMapping("/Produits")
    public List<Product> listeProduits() {
        return productDao.findAll();   // Délègue au DAO
    }

    @GetMapping("/Produits/{id}")
    public Product afficherUnProduit(@PathVariable int id) {
        return productDao.findById(id);
    }

    @PostMapping("/Produits")
    public Product ajouterProduit(@RequestBody Product product) {
        // @RequestBody = Jackson convertit le JSON reçu → objet Java Product
        return productDao.save(product);
    }
}

PARTIE 09 💾

JPA & la Persistance des Données

Faire survivre les données après un redémarrage — introduction à JPA et Hibernate.

Pourquoi une Base de Données ?

💀
Le Problème de l'Amnésie

Notre microservice avec ArrayList est "amnésique". À chaque redémarrage du serveur, TOUTES les données disparaissent. Les produits ajoutés via POST ? Perdus. En mémoire RAM = volatil = effacé à l'arrêt du programme.

🗄️
La solution : une armoire de classement externe

La base de données est comme une armoire de classement à l'extérieur du microservice. Même si le microservice s'éteint, les fiches (données) restent dans l'armoire. Au redémarrage, il retrouve tout son catalogue intact.

⚡ Le Problème : Java vs SQL ne parlent pas la même langue

C'est ce qu'on appelle le Déséquilibre d'Impédance Objet-Relationnel :

ProblèmeMonde JavaMonde SQL
NommagetargetDate (camelCase)target_date (snake_case)
RelationsList<Employee> dans une classeTables de jointure avec clés étrangères
HéritageFullTime extends EmployeeNon supporté nativement → 1 table aplatie + colonne discriminante
Unité de donnéesObjets avec comportements (méthodes)Lignes et colonnes — données uniquement

📜 Évolution des technologies (avant JPA)

TechnologieCe qu'on devait faireDifficulté
JDBC brutOuvrir connexion, écrire SQL à la main, parser le ResultSet colonne par colonne😫 Très pénible
Spring JDBC TemplateMoins de code, mais SQL toujours manuel dans des chaînes de caractères😐 Mieux, mais SQL risqué
MyBatisSQL dans des fichiers XML séparés, objets remplis automatiquement🙂 Propre, mais des milliers de fichiers SQL
JPA + HibernateZéro SQL → annote ta classe, JPA génère les requêtes😍 Très productif
Spring Data JPAEncore moins → juste une interface, Spring génère tout🤩 Presque zéro code

🔑 JPA vs Hibernate — Quelle différence ?

📋

JPA = La Spécification (les règles)

Définition : JPA (Java Persistence API) est un document de règles qui dit "voilà comment on doit sauvegarder des données en Java". Mais JPA seul ne "fait" rien.

Analogie : JPA = le Code de la route (les règles)

JPA définit les annotations : @Entity, @Id, @Column...

⚙️

Hibernate = L'Implémentation (le moteur)

Définition : Hibernate est le logiciel qui exécute réellement le travail de traduction entre Java et SQL.

Analogie : Hibernate = la Voiture (l'outil qui roule en suivant les règles)

Dans Spring Boot, on utilise JPA comme standard et Hibernate travaille en arrière-plan.

🏷️ Le Mapping ORM — Le concept central

📖 ORM = Object-Relational Mapping

Le Mapping est le processus de traduction automatique entre le monde Java et le monde SQL. Il établit la correspondance :

Monde JavaCorrespondanceMonde SQL
class Product→Table product
Une instance d'objet→Une ligne dans la table
Un attribut (private String nom)→Une colonne (NOM)
List<Order> orders dans Product→Clé étrangère + jointure

🏷️ Les annotations JPA essentielles

java model/Product.java — Transformé en Entité JPA 
@Entity
// ↑ OBLIGATOIRE : "Cette classe représente une table dans la BDD"
// Si la classe s'appelle Product → JPA gère une table nommée "product"

@Table(name = "product")
// ↑ OPTIONNEL : Spécifier le nom exact de la table SQL
// Utile si le nom de la classe diffère du nom de la table

public class Product {

    @Id
    // ↑ OBLIGATOIRE : Marque ce champ comme la CLÉ PRIMAIRE de la table

    @GeneratedValue(strategy = GenerationType.IDENTITY)
    // ↑ La valeur est générée AUTOMATIQUEMENT par la BDD (auto-incrément)
    // 1, 2, 3, 4... → plus besoin de gérer les IDs manuellement
    private int id;

    @Column(name = "nom")
    // ↑ OPTIONNEL : Lie cet attribut Java à la colonne SQL "nom"
    // Si tu ne mets pas @Column, JPA utilise le nom de l'attribut

    @Size(min = 3)
    // ↑ VALIDATION : Le nom doit avoir au moins 3 caractères
    private String nom;

    @Min(1)
    // ↑ VALIDATION : Le prix doit être au minimum 1
    private int prix;

    // constructeurs, getters, setters... (inchangés)
}

MéthodeActionSQL généréAnalogie

merge(obj)Mettre à jour un objet existantUPDATE ... SET ...Modifier le numéro d'un contact

📌 Résumé — Chapitre 9
  • ArrayList = données perdues au redémarrage → il faut une vraie BDD
  • JPA = la spécification (les règles). Hibernate = l'implémentation (le moteur)
  • ORM = le traducteur : classe Java ↔ table SQL, objet Java ↔ ligne SQL
  • @Entity = marque une classe comme table BDD, @Id = clé primaire
  • H2 = base de données légère en mémoire, parfaite pour le développement et les tests
PARTIE 10 🪄

Spring Data JPA — Zéro SQL, Zéro Implémentation

La magie ultime : une simple interface suffit pour accéder à toute la base de données.

Le Problème de l'EntityManager pur

Si ton projet a 50 entités (Produit, Commande, Paiement, Utilisateur...), tu dois créer 50 classes d'implémentation DAO avec les mêmes méthodes répétitives. Hundreds de lignes de copier-coller = source d'erreurs.

✨
La Solution Spring Data JPA

Tu déclares une simple interface qui hérite de JpaRepository. Spring génère automatiquement toute l'implémentation SQL au démarrage de l'application. Zéro classe d'implémentation à écrire !

La Pyramide de la Persistance

Ton Application

JPA (La norme / les règles)
Hibernate (Le moteur traducteur)
JDBC (Le tuyau physique)
Base de Données (H2 / MySQL / PostgreSQL...)

Plus on monte dans la pyramide, plus c'est simple à utiliser

📝 Étape 2 — La Transformation de ProductDao

    //              ↑ 1er paramètre : L'entité gérée (notre classe Product)
    //                              ↑ 2e paramètre : Le type de l'@Id (int → Integer)

    // Spring génère un "Proxy" au démarrage avec TOUT le code SQL
    // Tu obtiens GRATUITEMENT : findAll(), save(), deleteById(), count()...
}
// SUPPRIME ProductDaoImpl.java — elle n'est plus nécessaire !

🪄 Les requêtes générées par le nom de la méthode

Spring Data JPA peut générer le SQL uniquement à partir du nom de la méthode dans l'interface !

javadao/ProductDao.java — La magie des noms de méthodes
public interface ProductDao extends JpaRepository<Product, Integer> {

    // Déclare juste le nom — Spring écrit le SQL tout seul !

    List<Product> findByPrixGreaterThan(int prixLimit);
    // SQL généré : SELECT * FROM product WHERE prix > ?
    // Décomposition : findBy=SELECT, Prix=colonne, GreaterThan=opérateur(>)

    List<Product> findByPrixLessThan(int prixMax);
    // SQL généré : SELECT * FROM product WHERE prix < ?

    List<Product> findByNomContaining(String motCle);
    // SQL généré : SELECT * FROM product WHERE nom LIKE '%motCle%'

    List<Product> findByNomAndPrix(String nom, int prix);
    // SQL généré : SELECT * FROM product WHERE nom=? AND prix=?
}

📝 Requêtes personnalisées avec @Query (JPQL)

Pour les requêtes complexes qu'on ne peut pas exprimer par le nom d'une méthode :

java@Query avec JPQL
@Query("SELECT p FROM Product p WHERE p.prix > :prixLimit")
// ↑ JPQL : On utilise le nom de la CLASSE Java (Product), pas le nom de la table SQL !
// Hibernate traduit ensuite en SQL selon ta base de données (H2, MySQL, Oracle...)
// :prixLimit = paramètre nommé dans la requête JPQL

List<Product> findProduitsChers(@Param("prixLimit") int prix);
// ↑ @Param("prixLimit") = fait le lien entre ":prixLimit" dans JPQL et la variable Java "prix"
📡
JPQL vs SQL — Quelle différence ?

SQL pur : SELECT * FROM product WHERE prix > 300 (cible les tables SQL)
JPQL : SELECT p FROM Product p WHERE p.prix > 300 (cible les classes Java)
Avantage de JPQL : si tu changes de base de données (H2 → MySQL → Oracle), tu ne changes pas tes requêtes. Hibernate adapte le dialecte SQL automatiquement.

PARTIE 11 🏗️

Architecture 3 Couches & CRUD Complet

La structure professionnelle d'une application Spring Boot et toutes les opérations CRUD.

Les 3 Couches

🌐

Couche 1 — API Tier (Contrôleurs)

Annotation : @RestController

Mission : Écouter les requêtes HTTP, convertir JSON ↔ objets Java, valider les entrées, déléguer à la couche Service.

Règle d'or : Un contrôleur ne contient aucune logique métier. Pas de calculs, pas de règles de gestion — uniquement de la gestion HTTP.

🧠

Couche 2 — Service Tier (Logique métier)

Annotation : @Service

Mission : Appliquer les règles métier (calculer la TVA, vérifier les permissions, appliquer des remises).

Avantage : Cette couche est indépendante du Web — elle peut être utilisée par une API REST, une app mobile, ou une tâche automatique.

💾

Couche 3 — Persistence Tier (Données)

Composants : Classes @Entity + interfaces JpaRepository

Mission : Faire le pont entre Java et SQL. L'Entité représente la table. Le Repository effectue les opérations CRUD.

🎯 Pourquoi 3 couches ? Les 3 bénéfices

BénéficeExplication pratique
MaintenabilitéChanger de H2 à MySQL ? Tu ne modifies que la couche Persistence. Le Contrôleur et le Service ne bougent pas.
TestabilitéOn peut tester la logique métier (Service) sans démarrer de serveur web ni de base de données (grâce aux Mocks).
RéutilisabilitéPlusieurs Contrôleurs peuvent partager le même Service. Plusieurs Services peuvent partager le même Repository.

💻 CRUD Complet — Le Code Final

javaProductController.java — CRUD complet avec Spring Data JPA
@RestController
public class ProductController {

    private final ProductDao productDao;  // Spring injecte automatiquement
    public ProductController(ProductDao productDao) { this.productDao = productDao; }

    // ══════════════════════════════════════════════════
    // ✅ LIRE TOUS — GET /Produits
    // ══════════════════════════════════════════════════
    @GetMapping("/Produits")
    public List<Product> listeProduits() {
        return productDao.findAll();
        // Spring Data JPA génère : SELECT * FROM product;
    }

    // ══════════════════════════════════════════════════
    // ✅ LIRE UN — GET /Produits/{id}
    // ══════════════════════════════════════════════════
    @GetMapping("/Produits/{id}")
    public Product afficherUnProduit(@PathVariable int id) {
        Product p = productDao.findById(id).orElse(null);
        if (p == null) throw new ProduitIntrouvableException("Produit introuvable : " + id);
        return p;
    }

    // ══════════════════════════════════════════════════
    // ✅ CRÉER — POST /Produits
    // ══════════════════════════════════════════════════
    @PostMapping("/Produits")
    @ResponseStatus(HttpStatus.CREATED)  // Renvoie 201 au lieu de 200
    public Product ajouterProduit(@Valid @RequestBody Product product) {
        // @Valid : vérifie les règles @Size, @Min dans Product
        // Si invalide → Spring retourne automatiquement 400 Bad Request
        return productDao.save(product);
        // Spring Data JPA génère : INSERT INTO product (nom, prix) VALUES (?, ?)
    }

    // ══════════════════════════════════════════════════
    // ✅ MODIFIER — PUT /Produits
    // ══════════════════════════════════════════════════
    @PutMapping("/Produits")
    public Product updateProduit(@RequestBody Product product) {
        // .save() AVEC un ID existant = UPDATE (pas INSERT)
        // Spring Data JPA génère : UPDATE product SET nom=?, prix=? WHERE id=?
        return productDao.save(product);
    }

    // ══════════════════════════════════════════════════
    // ✅ SUPPRIMER — DELETE /Produits/{id}
    // ══════════════════════════════════════════════════
    @DeleteMapping("/Produits/{id}")
    public void supprimerProduit(@PathVariable int id) {
        productDao.deleteById(id);
        // Spring Data JPA génère : DELETE FROM product WHERE id=?
    }
}
PARTIE 12 🛡️

Gestion des Erreurs & Validation

Une API qui ment ou qui accepte n'importe quoi est une API dangereuse.

😱 Le Problème : une API qui ment !

🚨
200 OK avec une réponse vide = mensonge !

Si un client fait GET /Produits/999 et que le produit 999 n'existe pas, notre API répond 200 OK avec un corps vide. C'est faux ! 200 signifie "tout s'est bien passé". Mais rien n'a été trouvé. Pour une autre machine, c'est très trompeur et peut provoquer des bugs en cascade.

🏗️ Étape 1 — Créer une Exception personnalisée

javaweb/exceptions/ProduitIntrouvableException.java
package com.ecommerce.microcommerce.web.exceptions;

@ResponseStatus(HttpStatus.NOT_FOUND)
// ↑ MAGIE : Lie cette exception au code HTTP 404 Not Found
// Dès que cette exception est lancée, Spring change le statut de la réponse

public class ProduitIntrouvableException extends RuntimeException {
    // RuntimeException = erreur qui survient durant l'exécution (pas à la compilation)

    public ProduitIntrouvableException(String message) {
        super(message);
        // super() passe le message au parent (RuntimeException)
        // Spring le récupère pour l'afficher dans les logs et la réponse JSON
    }
}

Dans le Contrôleur, on l'utilise ainsi :

java
@GetMapping("/Produits/{id}")
public Product afficherUnProduit(@PathVariable int id) {
    Product p = productDao.findById(id).orElse(null);

    if (p == null) {
        // Le produit n'existe pas → on lance une exception
        // Spring intercepte → retourne 404 Not Found au client
        throw new ProduitIntrouvableException("Produit introuvable avec l'id : " + id);
    }

    return p;  // Produit trouvé → 200 OK + JSON
}

xmlpom.xml — Ajouter la dépendance validation
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-validation</artifactId>

@Entity
public class Product {

    @Id

    // ↑ Si le nom a moins de 3 chars → requête rejetée avec 400 Bad Request

    // ↑ Si le prix est 0 ou négatif → requête rejetée avec 400 Bad Request

📌 Résumé — Chapitre 12
  • Ne jamais retourner 200 OK quand une ressource n'existe pas — utiliser 404
  • Réserver le 500 aux vrais crashs de code, pas aux données manquantes
  • @ResponseStatus sur une exception lie l'exception à un code HTTP
  • @Valid active la vérification des annotations de validation dans le modèle
  • Annotations de validation : @Size, @Min, @NotNull, @Email...

Partie A — QCM (Questions à Choix Multiple)
Question 01

Quel est le principal problème de l'architecture monolithique pour les grands systèmes ?

  • A) Elle est trop simple à construire
  • B) Pour scaler une fonctionnalité, on doit dupliquer toute l'application
  • C) Elle nécessite toujours un serveur externe séparé
  • D) Elle ne peut être écrite qu'en Python
✅ Réponse : B — Dans un monolithe, si la Feature A reçoit beaucoup de trafic, tu dois dupliquer l'application ENTIÈRE (Features A + B + C), même si B et C n'ont pas besoin de scaling. C'est un gaspillage énorme de ressources. Les microservices résolvent ce problème en permettant de scaler uniquement le service concerné.
Question 02

Que combine exactement l'annotation @RestController ?

  • A) @Service + @Repository
  • B) @Controller + @ResponseBody
  • C) @Entity + @Table
  • D) @Component + @Configuration
✅ Réponse : B — @Controller permet de gérer les requêtes HTTP. @ResponseBody indique que la valeur de retour des méthodes doit être sérialisée directement en JSON et envoyée au client (pas de page HTML). @RestController = les deux en une seule annotation.
Question 03

Quelle annotation utilises-tu pour extraire une valeur depuis l'URL, par exemple /Produits/42 ?

  • A) @RequestParam
  • B) @RequestBody
  • C) @PathVariable
  • D) @RequestHeader
✅ Réponse : C — @PathVariable fait le pont entre la variable dans l'URL {id} et le paramètre de la méthode Java. Exemple : @GetMapping("/Produits/{id}") + @PathVariable int id → si l'URL est /Produits/42, alors id = 42.
Question 04

Quel est le rôle de Jackson dans une application Spring Boot ?

  • A) Convertit automatiquement les objets Java en JSON (et inversement)
  • B) C'est le serveur web intégré
  • C) Il gère la connexion à la base de données
  • D) Il s'occupe du routage HTTP
✅ Réponse : A — Jackson est une librairie de sérialisation/désérialisation. Il lit les Getters d'un objet Java pour produire du JSON, et appelle les Setters quand il parse du JSON entrant pour créer un objet Java. Il est inclus automatiquement via spring-boot-starter-web.
Question 05

Si tu appelles productDao.save(product) avec un produit dont l'ID existe déjà en base, que se passe-t-il ?

  • A) Une exception DuplicateKeyException est levée
  • B) Un nouveau produit est créé avec un nouvel ID
  • C) Le produit existant est mis à jour (génère un UPDATE)
  • D) L'ancien produit est supprimé et un nouveau est inséré
✅ Réponse : C — La méthode .save() de Spring Data JPA est intelligente. Si l'objet a un ID qui existe déjà en base → génère un UPDATE. Si l'ID est null ou nouveau → génère un INSERT. C'est pourquoi on utilise la même méthode pour créer ET modifier !
Question 06

Quel code HTTP doit retourner ton API quand un produit n'est pas trouvé en base de données ?

  • A) 200 OK avec un corps vide
  • B) 500 Internal Server Error
  • C) 404 Not Found
  • D) 204 No Content
✅ Réponse : C — 404 Not Found est le code standard HTTP pour "la ressource demandée n'existe pas". Retourner 200 avec un corps vide est trompeur. Retourner 500 est incorrect (le serveur n'a pas crashé, c'est juste que la donnée n'existe pas).
Question 07

Quelle est la relation exacte entre JPA et Hibernate ?

  • A) JPA est une implémentation de Hibernate
  • B) JPA est la spécification (les règles), Hibernate est une implémentation de ces règles
  • C) Ce sont exactement la même chose avec des noms différents
  • D) Hibernate est le successeur de JPA qui le remplace
✅ Réponse : B — JPA définit l'API et les annotations (@Entity, @Id...). Hibernate est le moteur réel qui lit ces annotations et génère le SQL. Analogie : JPA = Code de la Route (les règles) / Hibernate = la Voiture (l'outil qui suit les règles). Dans Spring Boot, on code avec les standards JPA mais c'est Hibernate qui travaille en arrière-plan.
Question 08

Qu'est-ce que @SpringBootApplication combine réellement ?

  • A) @RestController + @Service + @Repository
  • B) @Configuration + @EnableAutoConfiguration + @ComponentScan
  • C) @Entity + @Table + @Id
  • D) @GetMapping + @PostMapping + @DeleteMapping
✅ Réponse : B — @Configuration permet de définir des Beans de configuration. @EnableAutoConfiguration active la détection et configuration automatique des librairies. @ComponentScan dit à Spring de scanner les packages pour trouver tous les composants (@Controller, @Service, @Repository...).
Partie B — Questions ouvertes
Question 09 — Définition

Explique avec tes propres mots la règle "Database per Service" dans l'architecture microservices et pourquoi elle est fondamentale.

Réponse : Chaque microservice doit posséder sa propre base de données privée qu'aucun autre service ne peut accéder directement. Les services ne peuvent communiquer qu'à travers leurs APIs. Cette règle est fondamentale car elle garantit l'indépendance réelle des services : si le Service Produits change son schéma de BDD, le Service Commandes n'est pas affecté. Sans cette règle, les services deviennent couplés à travers la base de données partagée, ce qui recréé les mêmes problèmes que le monolithe.
Question 10 — Comparaison

Explique la différence entre @RequestBody et @PathVariable. Donne un exemple concret pour chacun.

@PathVariable : Extrait une valeur depuis l'URL elle-même. Exemple : GET /Produits/42 → @GetMapping("/Produits/{id}") + @PathVariable int id → id = 42. Utilisé pour identifier quelle ressource on veut.

@RequestBody : Lit le corps JSON de la requête et le convertit en objet Java via Jackson. Exemple : POST /Produits avec corps {"nom":"iPad","prix":800} → @RequestBody Product product → product.getNom() = "iPad". Utilisé pour envoyer des données complètes (création/modification).

Règle simple : @PathVariable = identifier une ressource dans l'URL. @RequestBody = envoyer des données dans le corps de la requête.
Question 11 — Trace d'exécution

Décris étape par étape ce qui se passe quand un client envoie POST /Produits avec le JSON {"nom":"MacBook","prix":2500}.

Étapes :
1. Le client envoie une requête HTTP POST avec le JSON dans le corps
2. Le Dispatcher Servlet de Spring reçoit la requête et cherche quelle méthode correspond au pattern POST /Produits
3. Il trouve la méthode ajouterProduit() dans ProductController
4. @RequestBody déclenche Jackson → il appelle setNom("MacBook") et setPrix(2500) sur un nouvel objet Product
5. @Valid vérifie les règles : nom ≥ 3 chars ✅, prix ≥ 1 ✅
6. productDao.save(product) est appelé
7. Spring Data JPA Proxy génère : INSERT INTO product (nom, prix) VALUES ('MacBook', 2500)
8. Hibernate exécute le SQL sur H2
9. Le produit est retourné avec son ID auto-généré
10. @ResponseStatus(HttpStatus.CREATED) → Spring envoie 201 Created + le JSON du produit créé
